    19→    19→    19→    19→## Overview
    20→    20→    20→    20→
    21→    21→    21→    21→Module Epsilon serves as the data caching layer of the Data Pipeline System, providing high-performance caching services to all pipeline modules. As a cross-cutting infrastructure component, Module Epsilon accelerates data access patterns, reduces load on external enrichment sources, and enables efficient replay of transformation operations through intelligent result caching.
    22→    22→    22→    22→
    23→    23→    23→    23→### Core Responsibilities
    24→    24→    24→    24→
    25→    25→    25→    25→**Enrichment Caching**: Caching lookup results from external data sources to minimize redundant API calls and database queries. The enrichment cache stores reference data retrieved by Module Beta during transformation, with configurable TTL policies based on data volatility.
    26→    26→    26→    26→
    27→    27→    27→    27→**Transformation Result Caching**: Maintaining cached copies of transformation outputs for records that may require reprocessing. This enables efficient replay without re-executing expensive transformation chains and supports idempotent operation patterns.
    28→    28→    28→    28→
    29→    29→    29→    29→**Inter-Module Buffer Caching**: Providing temporary storage buffers between pipeline stages to smooth throughput variations and handle backpressure scenarios. Buffer caches absorb burst loads from Module Alpha and regulate delivery rates to Module Gamma.
    30→    30→    30→    30→
    31→    31→    31→    31→**Reference Data Caching**: Storing static and semi-static reference datasets including lookup tables, validation rules, and configuration data that modules access repeatedly during processing.
    32→    32→    32→    32→
    33→    33→    33→    33→### Cache Hierarchy
    34→    34→    34→    34→
    35→    35→    35→    35→Module Epsilon implements a multi-tier cache hierarchy optimizing for both latency and capacity:
    36→    36→    36→    36→
    37→    37→    37→    37→```
    38→    38→    38→    38→+-------------------------------------------------------------------------+
    39→    39→    39→    39→|                      MODULE EPSILON CACHE HIERARCHY                      |

    41→    41→    41→    41→|                                                                          |
    42→    42→    42→    42→|  +------------------+     +------------------+     +------------------+  |
    43→    43→    43→    43→|  |    L1 CACHE      |     |    L2 CACHE      |     |    L3 CACHE      |  |
    44→    44→    44→    44→|  |   (In-Memory)    |     |  (Distributed)   |     |   (Persistent)   |  |
    45→    45→    45→    45→|  +------------------+     +------------------+     +------------------+  |

    61→    61→    61→    61→### Design Principles
    62→    62→    62→    62→
    63→    63→    63→    63→**Tiered Access Pattern**: Cache lookups proceed through tiers in order of latency, with L1 checked first, then L2, then L3. Cache writes may target specific tiers based on data characteristics and access patterns.
    64→    64→    64→    64→
    65→    65→    65→    65→**Write-Through Consistency**: For critical data, writes propagate through all cache tiers synchronously to maintain consistency. For performance-critical paths, write-behind patterns defer lower-tier updates.
    66→    66→    66→    66→
    67→    67→    67→    67→**Intelligent Eviction**: Each cache tier implements eviction policies appropriate to its characteristics. L1 uses LRU for hot data, L2 uses LFU for frequently accessed data, and L3 uses TTL-based expiration for durable storage.
    68→    68→    68→    68→
    69→    69→    69→    69→**Graceful Degradation**: Cache failures do not halt pipeline processing. When cache operations fail, modules fall back to source data access with appropriate performance implications logged and monitored.

    73→    73→    73→    73→## Cache Architecture
    74→    74→    74→    74→
    75→    75→    75→    75→The caching architecture consists of three tiers with distinct characteristics and purposes.
    76→    76→    76→    76→
    77→    77→    77→    77→### Tier 1: In-Memory Cache (L1)
    78→    78→    78→    78→
    79→    79→    79→    79→L1 provides ultra-low-latency access for the hottest data, storing cached entries in process memory with sub-millisecond access times.
    80→    80→    80→    80→
    81→    81→    81→    81→**Characteristics**:
    82→    82→    82→    82→- Access latency: <1ms
    83→    83→    83→    83→- Capacity: Configurable, default 1GB per process
    84→    84→    84→    84→- Scope: Single process, not shared across instances
    85→    85→    85→    85→- Persistence: None, volatile on restart
    86→    86→    86→    86→- Eviction: LRU with configurable max entries
    87→    87→    87→    87→
    88→    88→    88→    88→**Use Cases**:
    89→    89→    89→    89→- Frequently accessed enrichment lookups
    90→    90→    90→    90→- Hot transformation rule configurations
    91→    91→    91→    91→- Active session state
    92→    92→    92→    92→- Recently transformed record results
    93→    93→    93→    93→
    94→    94→    94→    94→**Implementation**: Lock-free concurrent hash map with segmented locks for write operations. Read operations are wait-free for maximum throughput.
    95→    95→    95→    95→
    96→    96→    96→    96→### Tier 2: Distributed Cache (L2)
    97→    97→    97→    97→
    98→    98→    98→    98→L2 provides cluster-wide shared caching with moderate latency, enabling cache sharing across pipeline instances.
    99→    99→    99→    99→
   100→   100→   100→   100→**Characteristics**:
   101→   101→   101→   101→- Access latency: 1-10ms
   102→   102→   102→   102→- Capacity: Configurable, default 50GB cluster-wide
   103→   103→   103→   103→- Scope: Shared across all pipeline instances
   104→   104→   104→   104→- Persistence: Optional write-ahead logging
   105→   105→   105→   105→- Eviction: LFU with TTL-based expiration
   106→   106→   106→   106→
   107→   107→   107→   107→**Use Cases**:
   108→   108→   108→   108→- Enrichment results shared across instances
   109→   109→   109→   109→- Lookup table caching
   110→   110→   110→   110→- Transformation profile configurations
   111→   111→   111→   111→- Cross-instance deduplication state
   112→   112→   112→   112→
   113→   113→   113→   113→**Partitioning**: Data partitioned by consistent hashing on cache keys. Partition count configurable, default 256 partitions. Rebalancing occurs automatically on node join/leave.
   114→   114→   114→   114→
   115→   115→   115→   115→**Replication**: Configurable replication factor (default 2) ensures data availability during node failures. Synchronous replication for critical data, asynchronous for performance-critical paths.
   116→   116→   116→   116→
   117→   117→   117→   117→### Tier 3: Persistent Cache (L3)
   118→   118→   118→   118→
   119→   119→   119→   119→L3 provides durable storage for cache entries that must survive restarts, storing data on disk with optional compression.
   120→   120→   120→   120→
   121→   121→   121→   121→**Characteristics**:
   122→   122→   122→   122→- Access latency: 10-50ms
   123→   123→   123→   123→- Capacity: Configurable, default 500GB per node
   124→   124→   124→   124→- Scope: Local to node with cluster coordination
   125→   125→   125→   125→- Persistence: Full durability with write-ahead logging
   126→   126→   126→   126→- Eviction: TTL-based with configurable retention
   127→   127→   127→   127→
   128→   128→   128→   128→**Use Cases**:
   129→   129→   129→   129→- Historical transformation results for replay
   130→   130→   130→   130→- Audit trail of enrichment lookups
   131→   131→   131→   131→- Long-lived reference data snapshots
   132→   132→   132→   132→- Recovery checkpoints
   133→   133→   133→   133→
   134→   134→   134→   134→**Storage Format**: LSM-tree based storage with block-level compression. Supports range queries for batch cache operations.
   135→   135→   135→   135→
   136→   136→   136→   136→### Cache Router
   137→   137→   137→   137→
   138→   138→   138→   138→The cache router determines which tier(s) to access for each operation based on cache policies, data characteristics, and current system state.
   139→   139→   139→   139→
   140→   140→   140→   140→**Routing Decisions**:
   141→   141→   141→   141→- Read operations: Check tiers in order L1 -> L2 -> L3 until hit
   142→   142→   142→   142→- Write operations: Write to tiers specified by cache policy
   143→   143→   143→   143→- Invalidation: Propagate to all tiers containing the entry
   144→   144→   144→   144→- Warm-up: Preload entries from L3 to L2/L1 on startup
   145→   145→   145→   145→
   146→   146→   146→   146→**Policy Engine**: Evaluates cache policies to determine tier routing, TTL assignment, and eviction priority for each cache operation.

   276→   276→   276→   276→## Cache Policies
   277→   277→   277→   277→
   278→   278→   278→   278→Module Epsilon implements a comprehensive policy framework governing cache behavior.
   279→   279→   279→   279→
   280→   280→   280→   280→### Policy Categories
   281→   281→   281→   281→
   282→   282→   282→   282→- **Lifecycle Policies**: Control entry creation, expiration, and removal
   283→   283→   283→   283→- **Tier Policies**: Determine which cache tiers store entries
   284→   284→   284→   284→- **Eviction Policies**: Select entries for removal under memory pressure
   285→   285→   285→   285→- **Consistency Policies**: Manage data consistency across tiers
   286→   286→   286→   286→- **Access Policies**: Control cache access patterns and rate limits
   287→   287→   287→   287→
   288→   288→   288→   288→### Standard Cache Policies
   289→   289→   289→   289→
   290→   290→   290→   290→#### Policy 1: TTL-Based Expiration
   291→   291→   291→   291→
   292→   292→   292→   292→Entries expire after a configurable time-to-live period.
   293→   293→   293→   293→
   294→   294→   294→   294→**Configuration**: `ttl_seconds` (integer), `ttl_jitter_percent` (float)
   295→   295→   295→   295→
   296→   296→   296→   296→```
   297→   297→   297→   297→ttl_seconds: 3600
   298→   298→   298→   298→ttl_jitter_percent: 10  # Add ±10% random jitter to prevent thundering herd
   299→   299→   299→   299→```
   300→   300→   300→   300→
   301→   301→   301→   301→#### Policy 2: LRU Eviction
   302→   302→   302→   302→
   303→   303→   303→   303→Least Recently Used entries evicted first when capacity reached.
   304→   304→   304→   304→
   305→   305→   305→   305→**Configuration**: `max_entries` (integer), `eviction_batch_size` (integer)
   306→   306→   306→   306→
   307→   307→   307→   307→```
   308→   308→   308→   308→max_entries: 100000
   309→   309→   309→   309→eviction_batch_size: 1000  # Evict in batches for efficiency
   310→   310→   310→   310→```
   311→   311→   311→   311→
   312→   312→   312→   312→#### Policy 3: LFU Eviction
   313→   313→   313→   313→
   314→   314→   314→   314→Least Frequently Used entries evicted first, favoring frequently accessed data.
   315→   315→   315→   315→
   316→   316→   316→   316→**Configuration**: `frequency_window_seconds` (integer), `min_frequency_threshold` (integer)
   317→   317→   317→   317→
   318→   318→   318→   318→```
   319→   319→   319→   319→frequency_window_seconds: 3600  # Track frequency over 1 hour
   320→   320→   320→   320→min_frequency_threshold: 5      # Entries with <5 accesses eligible for eviction
   321→   321→   321→   321→```
   322→   322→   322→   322→
   323→   323→   323→   323→#### Policy 4: Size-Based Eviction
   324→   324→   324→   324→
   325→   325→   325→   325→Large entries prioritized for eviction to maximize entry count.
   326→   326→   326→   326→
   327→   327→   327→   327→**Configuration**: `max_entry_size_bytes` (integer), `size_eviction_threshold` (float)
   328→   328→   328→   328→
   329→   329→   329→   329→```
   330→   330→   330→   330→max_entry_size_bytes: 1048576   # 1MB max per entry
   331→   331→   331→   331→size_eviction_threshold: 0.8    # Trigger eviction at 80% capacity
   332→   332→   332→   332→```
   333→   333→   333→   333→
   334→   334→   334→   334→#### Policy 5: Write-Through Consistency
   335→   335→   335→   335→
   336→   336→   336→   336→Writes propagate synchronously to all specified tiers before acknowledging.
   337→   337→   337→   337→
   338→   338→   338→   338→**Configuration**: `write_tiers` (list), `write_timeout_ms` (integer)
   339→   339→   339→   339→
   340→   340→   340→   340→```
   341→   341→   341→   341→write_tiers: [L1, L2]
   342→   342→   342→   342→write_timeout_ms: 5000
   343→   343→   343→   343→```
   344→   344→   344→   344→
   345→   345→   345→   345→#### Policy 6: Write-Behind Batching
   346→   346→   346→   346→
   347→   347→   347→   347→Writes acknowledged immediately, batched to lower tiers asynchronously.
   348→   348→   348→   348→
   349→   349→   349→   349→**Configuration**: `batch_size` (integer), `batch_interval_ms` (integer), `max_pending_writes` (integer)
   350→   350→   350→   350→
   351→   351→   351→   351→```
   352→   352→   352→   352→batch_size: 100
   353→   353→   353→   353→batch_interval_ms: 1000
   354→   354→   354→   354→max_pending_writes: 10000
   355→   355→   355→   355→```
   356→   356→   356→   356→
   357→   357→   357→   357→#### Policy 7: Read-Through Population
   358→   358→   358→   358→
   359→   359→   359→   359→Cache misses trigger automatic population from source data.
   360→   360→   360→   360→
   361→   361→   361→   361→**Configuration**: `source_timeout_ms` (integer), `populate_on_miss` (boolean)
   362→   362→   362→   362→
   363→   363→   363→   363→```
   364→   364→   364→   364→source_timeout_ms: 5000

   478→   478→   478→   478→### Policy Evaluation Order
   479→   479→   479→   479→
   480→   480→   480→   480→1. Access Rate Limiting (reject if exceeded)
   481→   481→   481→   481→2. Namespace Isolation (determine quota context)
   482→   482→   482→   482→3. Size-Based validation (reject oversized entries)
   483→   483→   483→   483→4. Tier routing (determine target tiers)
   484→   484→   484→   484→5. Compression (apply if threshold met)

   486→   486→   486→   486→7. TTL assignment (with jitter)
   487→   487→   487→   487→8. Replication (if enabled)
   488→   488→   488→   488→9. Eviction check (if capacity reached)

   492→   492→   492→   492→## Error Handling
   493→   493→   493→   493→
   494→   494→   494→   494→Module Epsilon implements comprehensive error handling following the principle of graceful degradation: cache failures should never halt pipeline processing.
   495→   495→   495→   495→
   496→   496→   496→   496→### Error Categories
   497→   497→   497→   497→
   498→   498→   498→   498→#### Cache Miss Errors
   499→   499→   499→   499→
   500→   500→   500→   500→Occur when requested entries are not found in any cache tier.
   501→   501→   501→   501→
   502→   502→   502→   502→**Common Causes**: Entry never cached, entry expired, entry evicted, key mismatch
   503→   503→   503→   503→
   504→   504→   504→   504→**Handling**: Return MISS status with null value. Caller responsible for fetching from source. Log miss for metrics. If `populate_on_miss` enabled, trigger async population.
   505→   505→   505→   505→
   506→   506→   506→   506→**CacheMissResult Structure**:
   507→   507→   507→   507→```
   508→   508→   508→   508→CacheMissResult:
   509→   509→   509→   509→  key: CacheKey
   510→   510→   510→   510→  tiers_checked: list<enum>       # Which tiers were checked
   511→   511→   511→   511→  miss_reason: enum               # NEVER_CACHED, EXPIRED, EVICTED
   512→   512→   512→   512→  lookup_latency_ms: float
   513→   513→   513→   513→  recommendation: enum            # FETCH_FROM_SOURCE, RETRY, USE_STALE
   514→   514→   514→   514→```
   515→   515→   515→   515→
   516→   516→   516→   516→**Metrics**: `epsilon_cache_misses_total`, `epsilon_miss_rate_by_namespace`
   517→   517→   517→   517→
   518→   518→   518→   518→#### Stale Data Errors
   519→   519→   519→   519→
   520→   520→   520→   520→Occur when cached data may be outdated but still accessible.
   521→   521→   521→   521→
   522→   522→   522→   522→**Common Causes**: Source data changed, invalidation delayed, refresh failed
   523→   523→   523→   523→
   524→   524→   524→   524→**Handling**: If `stale_while_revalidate` enabled, return stale value with staleness flag. Trigger async refresh. If stale data unacceptable, return MISS. Log staleness for monitoring.
   525→   525→   525→   525→
   526→   526→   526→   526→**StaleDataWarning Structure**:
   527→   527→   527→   527→```
   528→   528→   528→   528→StaleDataWarning:
   529→   529→   529→   529→  key: CacheKey
   530→   530→   530→   530→  cached_at: datetime
   531→   531→   531→   531→  expected_ttl: integer
   532→   532→   532→   532→  actual_age_seconds: integer
   533→   533→   533→   533→  staleness_reason: enum          # REFRESH_FAILED, INVALIDATION_PENDING
   534→   534→   534→   534→  stale_value: bytes              # The stale cached value
   535→   535→   535→   535→```
   536→   536→   536→   536→
   537→   537→   537→   537→**Configuration**: `STALE_WHILE_REVALIDATE_ENABLED`, `MAX_STALE_AGE_SECONDS`
   538→   538→   538→   538→
   539→   539→   539→   539→#### Cache Corruption Errors
   540→   540→   540→   540→
   541→   541→   541→   541→Occur when cached data fails integrity validation.
   542→   542→   542→   542→
   543→   543→   543→   543→**Common Causes**: Checksum mismatch, deserialization failure, storage corruption, version incompatibility
   544→   544→   544→   544→
   545→   545→   545→   545→**Handling**: Immediately invalidate corrupted entry across all tiers. Log corruption event with full context. Increment corruption counter. If corruption rate exceeds threshold, trigger cache health alert.
   546→   546→   546→   546→
   547→   547→   547→   547→**CacheCorruptionError Structure**:
   548→   548→   548→   548→```
   549→   549→   549→   549→CacheCorruptionError:
   550→   550→   550→   550→  key: CacheKey
   551→   551→   551→   551→  tier: enum
   552→   552→   552→   552→  corruption_type: enum           # CHECKSUM_MISMATCH, DESERIALIZE_FAIL, TRUNCATED
   553→   553→   553→   553→  expected_checksum: string
   554→   554→   554→   554→  actual_checksum: string
   555→   555→   555→   555→  entry_size: integer
   556→   556→   556→   556→  detection_timestamp: datetime
   557→   557→   557→   557→```
   558→   558→   558→   558→
   559→   559→   559→   559→**Recovery**: `INVALIDATE_AND_REFETCH`, automatic corruption recovery enabled by default
   560→   560→   560→   560→
   561→   561→   561→   561→**Metrics**: `epsilon_corruption_events_total`, `epsilon_corruption_by_tier`
   562→   562→   562→   562→
   563→   563→   563→   563→#### Distributed Sync Errors
   564→   564→   564→   564→
   565→   565→   565→   565→Occur when cache operations fail to propagate across cluster nodes.
   566→   566→   566→   566→
   567→   567→   567→   567→**Common Causes**: Network partition, node failure, replication timeout, consistency conflict
   568→   568→   568→   568→
   569→   569→   569→   569→**Handling**: For write operations, apply configured consistency policy:
   570→   570→   570→   570→- **SYNC_GUARANTEED**: Retry until success or timeout, then fail operation
   571→   571→   571→   571→- **ASYNC_BEST_EFFORT**: Log failure, continue operation, schedule retry
   572→   572→   572→   572→- **EVENTUAL**: Queue for background reconciliation
   573→   573→   573→   573→
   574→   574→   574→   574→**DistributedSyncError Structure**:
   575→   575→   575→   575→```
   576→   576→   576→   576→DistributedSyncError:
   577→   577→   577→   577→  operation_id: string
   578→   578→   578→   578→  operation_type: enum
   579→   579→   579→   579→  source_node: string
   580→   580→   580→   580→  target_nodes: list<string>
   581→   581→   581→   581→  failed_nodes: list<string>
   582→   582→   582→   582→  error_code: string              # e.g., "SYNC_001_PARTITION"
   583→   583→   583→   583→  retry_count: integer
   584→   584→   584→   584→  is_recoverable: boolean
   585→   585→   585→   585→```
   586→   586→   586→   586→
   587→   587→   587→   587→**Conflict Resolution**: Last-write-wins with vector clock tiebreaker
   588→   588→   588→   588→
   589→   589→   589→   589→**Metrics**: `epsilon_sync_failures_total`, `epsilon_partition_events`, `epsilon_conflict_resolutions`
   590→   590→   590→   590→
   591→   591→   591→   591→#### Capacity Exhaustion Errors
   592→   592→   592→   592→
   593→   593→   593→   593→Occur when cache tiers reach capacity limits.
   594→   594→   594→   594→
   595→   595→   595→   595→**Common Causes**: Eviction not keeping pace with writes, unusually large entries, configuration undersized
   596→   596→   596→   596→
   597→   597→   597→   597→**Handling**: Trigger emergency eviction of lowest priority entries. If still at capacity, reject new writes with CAPACITY_EXCEEDED. Log capacity event. If sustained, alert operators.
   598→   598→   598→   598→
   599→   599→   599→   599→**CapacityExhaustedError Structure**:
   600→   600→   600→   600→```
   601→   601→   601→   601→CapacityExhaustedError:
   602→   602→   602→   602→  tier: enum
   603→   603→   603→   603→  current_capacity_bytes: integer
   604→   604→   604→   604→  max_capacity_bytes: integer
   605→   605→   605→   605→  current_entries: integer
   606→   606→   606→   606→  max_entries: integer
   607→   607→   607→   607→  eviction_candidates: integer
   608→   608→   608→   608→  pressure_level: enum            # WARNING, CRITICAL, EXHAUSTED
   609→   609→   609→   609→```
   610→   610→   610→   610→
   611→   611→   611→   611→**Emergency Actions**: `AGGRESSIVE_EVICTION`, `REJECT_LOW_PRIORITY`, `DISABLE_TIER`
   612→   612→   612→   612→
   613→   613→   613→   613→**Configuration**: `CAPACITY_WARNING_THRESHOLD`, `CAPACITY_CRITICAL_THRESHOLD`
   614→   614→   614→   614→
   615→   615→   615→   615→#### Connection Pool Exhaustion
   616→   616→   616→   616→
   617→   617→   617→   617→Occur when connections to distributed cache nodes are exhausted.
   618→   618→   618→   618→
   619→   619→   619→   619→**Common Causes**: Connection leak, high request rate, slow network, node unresponsive
   620→   620→   620→   620→
   621→   621→   621→   621→**Handling**: Queue requests up to `MAX_QUEUED_REQUESTS`. Timeout queued requests after `QUEUE_TIMEOUT_MS`. Circuit break to failing nodes. Fall back to local cache if distributed unavailable.
   622→   622→   622→   622→
   623→   623→   623→   623→**Recovery**: Automatic connection recycling, exponential backoff on reconnect
   624→   624→   624→   624→
   625→   625→   625→   625→### Error Queue Management
