    18→## Overview
    19→
    20→The Integration Layer defines the communication protocols, message formats, and coordination mechanisms that enable reliable data transfer between the modules of the Data Pipeline System. This specification establishes the authoritative contracts governing inter-module communication.
    21→
    22→### Purpose and Scope
    23→
    24→The Integration Layer serves as the connective tissue binding Module Alpha (Data Ingestion), Module Beta (Data Transformation), Module Gamma (Data Output), Module Epsilon (Data Caching), and Module Phi (Pipeline Orchestration) into a cohesive processing pipeline. Its responsibilities include:
    25→
    26→**Protocol Definition**: Establishing the exact sequence of operations, message formats, and acknowledgment patterns for each inter-module handoff.
    27→
    28→**Flow Control**: Managing throughput variations between modules through back-pressure signaling, buffer management, and adaptive rate limiting.
    29→
    30→**Error Coordination**: Propagating error information across module boundaries in a structured format that preserves diagnostic context.
    31→
    32→**Health Coordination**: Aggregating health status from all modules to present a unified view of pipeline health.
    33→
    34→### Module Interaction Overview
    35→
    36→```
    37→┌─────────────────────────────────────────────────────────────────┐
    38→│                   INTEGRATION LAYER OVERVIEW                     │
    39→├─────────────────────────────────────────────────────────────────┤
    40→│    ┌──────────────────┐                                          │
    41→│    │   MODULE ALPHA   │  • Source Adapters  • Validation         │
    42→│    │  Data Ingestion  │  • Input Parsing    • Buffering          │
    43→│    └────────┬─────────┘                                          │
    44→│             │  ALPHA-BETA HANDOFF                                │
    45→│             │  ├─ Batch Transfer    ├─ Acknowledgment            │
    46→│             ▼  └─ Back-Pressure                                  │
    47→│    ┌──────────────────┐                                          │
    48→│    │   MODULE BETA    │  • Schema Mapping   • Enrichment         │
    49→│    │ Transformation   │  • Field Transform  • Quality Scoring    │
    50→│    └────────┬─────────┘                                          │
    51→│             │  BETA-GAMMA HANDOFF                                │
    52→│             │  ├─ Batch Transfer    ├─ Acknowledgment            │
    53→│             ▼  └─ Back-Pressure                                  │
    54→│    ┌──────────────────┐                                          │
    55→│    │   MODULE GAMMA   │  • Format Render    • Ack Handling       │
    56→│    │   Data Output    │  • Delivery Route   • Dead Letter Mgmt   │
    57→│    └──────────────────┘                                          │
    58→└─────────────────────────────────────────────────────────────────┘
    59→```

    63→## Message Formats
    64→
    65→The Integration Layer defines standardized message formats for all inter-module communication.
    66→
    67→### Common Message Elements
    68→
    69→#### MessageHeader
    70→
    71→```
    72→MessageHeader:
    73→  message_id: string            # UUID v4 uniquely identifying this message
    74→  correlation_id: string        # Links related messages across the pipeline
    75→  source_module: enum           # ALPHA, BETA, GAMMA
    76→  target_module: enum           # ALPHA, BETA, GAMMA
    77→  message_type: enum            # BATCH_TRANSFER, ACKNOWLEDGMENT, BACKPRESSURE
    78→  timestamp: datetime           # ISO 8601 with microsecond precision, UTC
    79→  protocol_version: string      # Semantic version (e.g., "1.0.0")
    80→```
    81→
    82→#### BatchMetadata
    83→
    84→```
    85→BatchMetadata:
    86→  batch_id: string              # UUID v4 uniquely identifying this batch
    87→  sequence_number: integer      # Monotonically increasing per source module
    88→  record_count: integer         # Number of records in batch
    89→  checksum: string              # SHA-256 of batch contents
    90→  created_at: datetime          # When batch assembly completed
    91→```
    92→
    93→#### AcknowledgmentPayload
    94→
    95→```
    96→AcknowledgmentPayload:
    97→  batch_id: string              # Batch being acknowledged
    98→  status: enum                  # RECEIVED, PROCESSED, REJECTED
    99→  records_accepted: integer     # Successfully processed record count
   100→  records_rejected: integer     # Failed record count
   101→  processing_duration_ms: integer
   102→```
   103→

   150→## Alpha-to-Beta Protocol
   151→
   152→The Alpha-to-Beta protocol governs the transfer of validated records from the ingestion layer to the transformation layer.
   153→
   154→### Protocol Flow
   155→
   156→```
   157→MODULE ALPHA                                MODULE BETA
   158→────────────                                ───────────
   159→     │  1. Batch Assembly                        │
   160→     │     (buffer threshold or timeout)         │
   161→     │                                           │
   162→     │────── AlphaBetaBatchTransfer ────────────▶│
   163→     │                                           │
   164→     │                               2. Integrity Check
   165→     │◀───── Acknowledgment (RECEIVED) ─────────│
   166→     │                                           │
   167→     │  3. Buffer Release            4. Process Batch
   168→     │                                           │
   169→     │◀───── Acknowledgment (PROCESSED) ────────│
   170→```
   171→
   172→### Batch Assembly Rules
   173→
   174→**Size Threshold**: Transfer when `ALPHA_BETA_BATCH_SIZE` records accumulate.
   175→
   176→**Time Threshold**: Transfer after `ALPHA_BETA_FLUSH_TIMEOUT_MS` regardless of batch size.
   177→
   178→**Priority Override**: CRITICAL priority records trigger immediate transfer.
   179→
   180→### Integrity Verification
   181→
   182→Module Beta performs on receipt:
   183→1. **Checksum Validation**: Recalculate SHA-256 and compare with `metadata.checksum`
   184→2. **Record Count Validation**: Verify `records.length` matches `metadata.record_count`
   185→3. **Duplicate Detection**: Check `batch_id` against recently received batches
   186→
   187→### Back-Pressure Signaling
   188→
   189→```
   190→BackPressureSignal:
   191→  header: MessageHeader
   192→  signal_type: enum             # ACTIVATE, DEACTIVATE
   193→  current_queue_depth: integer
   194→  recommended_delay_ms: integer
   195→```
   196→
   197→### Retry Behavior
   198→
   199→If acknowledgment is not received within `ALPHA_BETA_ACK_TIMEOUT_MS`:
   200→1. Retransmit after `ALPHA_BETA_RETRY_DELAY_MS`
   201→2. Apply exponential backoff up to `ALPHA_BETA_MAX_RETRIES`
   202→3. Route to error queue after exhaustion

   206→## Beta-to-Gamma Protocol
   207→
   208→The Beta-to-Gamma protocol governs the transfer of transformed records from the transformation layer to the output layer.
   209→
   210→### Protocol Flow
   211→
   212→```
   213→MODULE BETA                                 MODULE GAMMA
   214→───────────                                 ────────────
   215→     │  1. Batch Assembly                        │
   216→     │     (quality-filtered)                    │
   217→     │                                           │
   218→     │────── BetaGammaBatchTransfer ────────────▶│
   219→     │                                           │
   220→     │                               2. Integrity Check
   221→     │◀───── Acknowledgment (RECEIVED) ─────────│
   222→     │                                           │
   223→     │  3. Buffer Release            4. Route & Deliver
   224→     │                                           │
   225→     │◀───── Acknowledgment (PROCESSED) ────────│
   226→     │                                           │
   227→     │                               5. Delivery Confirmation
   228→     │◀───── DeliveryConfirmation ──────────────│
   229→```
   230→
   231→### Delivery Confirmation

   234→DeliveryConfirmation:

   236→  batch_id: string
   237→  delivery_results: list<RecordDeliveryResult>

   241→RecordDeliveryResult:
   242→  record_id: string

   250→## Error Propagation
   251→
   252→The Integration Layer implements a comprehensive error propagation framework ensuring failures are visible, actionable, and recoverable across all module boundaries.
   253→
   254→### Error Classification Framework
   255→
   256→#### Severity Levels
   257→
   258→**CRITICAL**: Systemic failure requiring immediate intervention. Examples: persistent connectivity loss, resource exhaustion, data corruption. Critical errors trigger alerts and may halt processing.
   259→
   260→**ERROR**: Failures preventing processing of specific records or batches without systemic failure. Examples: validation failures, transformation errors, delivery rejections.
   261→
   262→**WARNING**: Conditions that may indicate problems but do not prevent processing. Examples: elevated retry rates, degraded enrichment sources.
   263→
   264→#### Recoverability Classification
   265→
   266→**TRANSIENT**: May resolve through retry. Examples: network timeouts, temporary service unavailability, rate limiting. The Integration Layer applies retry logic with exponential backoff.
   267→
   268→**PERMANENT**: Will not resolve through retry. Examples: schema violations, invalid data formats, authorization failures. Bypass retry logic and proceed to error handling.
   269→
   270→#### Scope Classification
   271→
   272→**RECORD-LEVEL**: Affects individual records. Other records proceed normally.
   273→**BATCH-LEVEL**: Affects entire batches. All records in batch impacted.
   274→**MODULE-LEVEL**: Affects entire module operation.
   275→**PIPELINE-LEVEL**: Affects entire pipeline. All data flow halts.
   276→
   277→### Error Message Structure
   278→
   279→```
   280→IntegrationError:
   281→  error_id: string              # UUID v4 uniquely identifying this error
   282→  correlation_id: string        # Links to originating message/batch
   283→  timestamp: datetime           # When error was detected
   284→  source_module: enum           # ALPHA, BETA, GAMMA
   285→  severity: enum                # CRITICAL, ERROR, WARNING
   286→  recoverability: enum          # TRANSIENT, PERMANENT, UNKNOWN
   287→  scope: enum                   # RECORD, BATCH, MODULE, PIPELINE
   288→  error_code: string            # Hierarchical code (e.g., "INT_ALPHA_001")
   289→  error_message: string         # Human-readable description
   290→  context: ErrorContext
   291→  retry_eligible: boolean
   292→```
   293→
   294→```
   295→ErrorContext:
   296→  record_id: string             # Affected record (if record-level)
   297→  batch_id: string              # Affected batch (if batch-level)
   298→  field_name: string            # Affected field (if applicable)
   299→  expected_value: string
   300→  actual_value: string
   301→  upstream_errors: list<string> # Related upstream error IDs
   302→```
   303→
   304→### Module Alpha Error Handling Integration
   305→
   306→Module Alpha generates errors propagating through the Integration Layer. For complete specification, see `module-alpha.md` Section 5.
   307→
   308→**Connection Errors**: When source adapters cannot establish connectivity, Alpha generates MODULE-LEVEL errors. The Integration Layer notifies downstream modules to expect reduced throughput.
   309→
   310→**Parse Errors**: When input parsing fails, Alpha generates RECORD-LEVEL errors. Failed records are excluded from batches with error details in batch metadata.
   311→
   312→**Validation Errors**: When records fail validation rules, Alpha generates RECORD-LEVEL errors. The Integration Layer propagates summaries to Module Beta through `validation_summary`.
   313→
   314→**Buffer Overflow**: When Alpha's buffer reaches capacity without acknowledgment, Alpha generates MODULE-LEVEL errors triggering back-pressure protocols.
   315→
   316→### Module Beta Error Handling Integration
   317→
   318→Module Beta generates errors during transformation requiring Integration Layer coordination. For complete specification, see `module-beta.md` Section 5.
   319→
   320→**Mapping Errors**: When schema mapping fails, Beta generates RECORD-LEVEL errors. The Integration Layer ensures failed records are excluded from downstream batches.
   321→
   322→**Transformation Errors**: When field transformations produce invalid results, Beta generates RECORD-LEVEL errors preserved in the transformation audit trail.
   323→
   324→**Enrichment Errors**: When external enrichment sources are unavailable, Beta generates errors with severity per configuration. The Integration Layer coordinates circuit breaker state.
   325→
   326→**Quality Score Failures**: When records fall below thresholds, Beta generates WARNING-level errors. The Integration Layer tracks quality degradation trends.
   327→
   328→### Module Gamma Error Handling Integration
   329→
   330→Module Gamma generates errors during output processing completing the error propagation chain. For complete specification, see `module-gamma.md` Section 6.
   331→
   332→**Rendering Errors**: When format rendering fails, Gamma generates RECORD-LEVEL errors. The Integration Layer ensures failures are tracked against source records.
   333→
   334→**Routing Errors**: When no valid destination can be determined, Gamma generates RECORD-LEVEL errors. The Integration Layer may trigger re-evaluation of upstream rules.
   335→
   336→**Delivery Errors**: When destination delivery fails, Gamma generates errors with severity based on retry status. The Integration Layer coordinates retry timing and circuit breaker state.
   337→
   338→**Acknowledgment Errors**: When acknowledgments timeout or fail, Gamma generates WARNING or ERROR level issues. The Integration Layer ensures ambiguous states are tracked for reconciliation.
   339→
   340→### Cross-Module Error Propagation
   341→
   342→#### Downstream Propagation
   343→
   344→```
   345→BatchErrorSummary:
   346→  upstream_errors: integer
   347→  error_breakdown: map<string, integer>
   348→  affected_record_ids: list<string>
   349→```
   350→
   351→Downstream modules use this to skip failed records and adjust expectations.
   352→
   353→#### Upstream Propagation
   354→
   355→```
   356→ErrorNotification:
   357→  header: MessageHeader
   358→  errors: list<IntegrationError>
   359→  action_required: enum         # NONE, THROTTLE, HALT, INVESTIGATE
   360→```
   361→
   362→Upstream modules use this to adjust production rates and trigger alerts.
   363→
   364→### Circuit Breaker Coordination
   365→
   366→```
   367→CircuitBreakerState:
   368→  circuit_id: string
   369→  state: enum                   # CLOSED, OPEN, HALF_OPEN
   370→  failure_count: integer
   371→  last_failure: datetime
   372→  next_attempt: datetime
   373→```
   374→
   375→**CLOSED**: Normal operation. Failures increment counter.
   376→**OPEN**: Requests immediately rejected. Timer until half-open.
   377→**HALF_OPEN**: Single test request allowed.
   378→
   379→```
   380→CircuitStateChange:
   381→  header: MessageHeader
   382→  circuit_id: string
   383→  previous_state: enum
   384→  new_state: enum
   385→  affected_modules: list<enum>
   386→```
   387→
   388→### Error Recovery Procedures
   389→
   390→**Transient Network Errors**: Retry with exponential backoff. After exhaustion, reclassify as permanent.
   391→
   392→**Validation Failures**: No retry. Route to error queue with diagnostic context.
   393→
   394→**Resource Exhaustion**: Trigger back-pressure. Clear caches. Resume gradually.
   395→
   396→**Module Unavailability**: Open circuit breaker. Queue pending work. Alert operators.
   397→
   398→**Data Corruption**: Halt affected processing. Generate CRITICAL alert. Require intervention.
   399→
   400→### Error Metrics
   401→
   402→**Counters**: `integration_errors_total`, `integration_retries_total`, `integration_circuit_breaker_trips_total`
   403→
   404→**Gauges**: `integration_error_queue_depth`, `integration_circuit_breaker_state`
   405→
   406→**Histograms**: `integration_error_recovery_duration_seconds`

   423→HealthCheckResponse:
   424→  header: MessageHeader

   438→### Pipeline Health Status
   439→
   440→```
   441→PipelineHealthStatus:
   442→  overall_status: enum          # HEALTHY, DEGRADED, UNHEALTHY
   443→  module_statuses: map<enum, ModuleHealth>
   444→  active_circuits_open: integer
   445→  error_rate_1m: float
   446→  throughput_records_per_second: float
   447→```
   448→
   449→Status determination:
   450→- **HEALTHY**: All modules healthy, no circuits open, error rate below threshold
   451→- **DEGRADED**: Any module degraded OR any circuit open OR elevated error rate
   452→- **UNHEALTHY**: Any module unhealthy OR multiple circuits open

   471→## Module Epsilon Integration
   472→
   473→Module Epsilon provides caching services to all pipeline modules through the Integration Layer.
   474→
   475→### Cache Access Protocol
   476→
   477→All modules access the cache through standardized cache operations:
   478→
   479→```
   480→CacheAccessRequest:
   481→  header: MessageHeader
   482→  operation: enum              # GET, PUT, INVALIDATE
   483→  cache_key: string
   484→  cache_namespace: string      # e.g., "enrichment", "transformation"
   485→  value: bytes                 # For PUT operations
   486→  ttl_seconds: integer         # Optional TTL override
   487→```
   488→
   489→```
   490→CacheAccessResponse:
   491→  header: MessageHeader
   492→  status: enum                 # HIT, MISS, ERROR
   493→  value: bytes                 # For successful GET
   494→  source_tier: enum            # L1, L2, L3
   495→  latency_ms: float
   496→```
   497→
   498→### Cache Warm-Up Coordination
   499→
   500→Before batch processing, modules can request cache warm-up:
   501→
   502→```
   503→CacheWarmUpRequest:
   504→  header: MessageHeader
   505→  cache_namespace: string
   506→  key_patterns: list<string>
   507→  priority: enum               # NORMAL, HIGH
   508→```
   509→
   510→### Cache Health Integration
   511→
   512→Cache health is included in pipeline health status. See `module-epsilon.md` for complete cache specification.
